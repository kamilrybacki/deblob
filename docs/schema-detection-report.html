Lane 1 — deterministic structural detection

Before any model runs, Deblob fingerprints each event and retrieves the nearest known schema families by structural distance. This is the workhorse: fast, exact, and where most correct detections come from.

Read: for 92% of events the right family is already among the top three candidates on structure alone. The remaining 7.7% never surface the gold family — a retrieval ceiling no amount of model tuning can fix, so it is gated and reported separately.

Lane 2 — the trust gate: coverage vs. safety

A detector that decides everything will merge unrelated schemas. Deblob's gate holds a hard line — never merge on a guess — trading coverage for safety. Each arm below sees the same 96 events; only the decider changes.

gated (safe) ungated SLM retrieval only

A0 — retrieval only takes every top-1: 100% coverage, but 39 false merges in 96. Reckless.

The gate (A1 / B1 / B2) accepts only corroborated, unambiguous, high-margin matches: coverage falls to 54%, false merges to 1 (95% upper bound 10%). Everything else is handed to a human, safely.

The gate blocks 61% of retrieval's raw errors before they can merge.

The honest finding — a zero-shot model earns nothing

B1 ≈ B2 ≈ A1

Behind the gate, a zero-shot / mock SLM (B1) lands on the exact same coverage and false-merge count as pure deterministic top-1 (B2) and the deterministic policy (A1): 54.17% coverage, 1 false merge. The redundancy ablation is unambiguous — an un-tuned model adds no safe coverage. McNemar isn't significant; the paired-bootstrap accuracy delta straddles zero. Raw model accuracy alone does not justify the lane.

This is the whole reason the continual-learning loop exists. The intelligence lane only pays for itself once the model actually learns Deblob's decision — which is what the next section measures.

Lane 2, fine-tuned — the model learns the task

From 0% on every metric, the tiny model reaches 96% valid output and gets the decision right 46% of the time on families it never trained on. The schema_id is a 50-character opaque hash the model should not memorize — that is precisely what Lane 1's retrieval supplies and the gate corroborates.

Read it honestly

One round on 320 examples taught format + the majority class: the model now emits valid tool-calls and says match_schema for same-family events — but it over-predicts match (misses all new-family and abstain cases), often gets relation wrong, and hallucinates the schema_id. So the 46% is real recognition of same-family structure, not yet discrimination. This is expected at 0.5B / 320 examples — and it is why the raw SLM is never trusted alone: retrieval supplies identity, the gate rejects the over-matching. More data + decision-weighted loss are the next levers.

The pattern: the fine-tuned model is good at recognizing same-family events (the top row) and — encouragingly — got the incompatible-similarity safety case fully right, but it over-matches on genuinely-new and low-evidence events. Every one of those over-matches is caught downstream by the trust gate (rank, margin, observation floor, deterministic corroboration) — which is exactly why a still-imperfect model can be used without ever creating a false merge.

What's measured, what's next

Real & measured

  • Deterministic retrieval + gate arms on a ground-truth-by-construction corpus (n=96).
  • Real LoRA fine-tune on Modal T4, scored on family-held-out data (0→96% / 0→46%).
  • No-false-merge invariant is separately proven exhaustively in the test suite.

Honest caveats & next levers

  • The arm table's SLM (B0/B1) is a mock stand-in — real adapters wire in next.
  • Fine-tune was one small round; more data + decision-weighted loss should lift the 46%.
  • FunctionGemma-270M (lighter, FC-specialized) is gated — a follow-up once a token exists.
  • The loop's job: turn 46% into growing safe coverage, round over round.
