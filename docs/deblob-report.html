Deblob · technical report

How Deblob works, and how the experiment went

A Rust schema-tagging service for event streams: recognize which schema an event belongs to — fast and deterministically — and discover genuinely new schemas without ever wrongly merging two different ones. This is what it does, and what a real fine-tune of its intelligence lane proved.

github.com/kamilrybacki/deblob 2026-07-17 real hook proven end-to-end

In one paragraph

Deblob is built as two lanes: a deterministic hot path that does the fast, certain work, and a small-language-model (SLM) discovery lane behind a trust gate that structurally prevents an unproven model decision from ever being applied. A continual-learning loop lets the SLM improve from human decisions under strict governance — a bad model can't corrupt identity or auto-deploy. The recent work fine-tuned that SLM lane on real hardware and proved the production training hook end-to-end: a feedback snapshot went through the deployed trainer, produced a governed adapter, and that exact adapter — loaded back — reproduced the evaluation numbers.

The honest verdict

An advanced deterministic schema-tagging prototype with a credible, safety-oriented semantic research lane — pre-alpha, unreleased. Its core contribution isn't that a 0.5B model reaches a particular accuracy; it's that model improvement raises coverage without transferring identity authority away from deterministic policy.

How it works — the two lanes

deterministic · hot path

Structure decides the certain cases

Parse (fuzz-hardened) → canonical fingerprint → monoid profile → structural neighbor retrieval. Exactly-once Kafka relay (chaos-tested, tuned 14→598 rec/s); Redis vault (atomic, immutable, health-gated). Fast, deterministic — most correct detections come from here (retrieval recall@3 ≈ 92%).

SLM · cold discovery

A model handles the subtle middle

For the ambiguous cases (same-structure-different-meaning, drift, renamed fields), a small model proposes a 3-way decision — match_schema, new_candidate, or abstain — and never runs on the latency-critical path. It can fail closed without taking down the relay.

The trust gate — the load-bearing safety property

An SLM proposal is applied only if a deterministic gate corroborates it — rank==1, distance ≤ 0.15, margin ≥ 0.10, observations ≥ 20, deterministic-compat, no redaction collision — with no model confidence in the decision. The gate structurally prevents uncorroborated model outputs from being applied. The honest safety claim: the gate prevented application and produced zero observed false-merges over a stated, externally-labeled sample — with a sample size and an upper confidence bound (the model registry already enforces Wilson bounds, per-family floors, and non-inferiority).

The continual-learning control plane

Stronger than "collect feedback + periodically fine-tune": immutable provenance-bearing feedback, family + near-duplicate partitioning, reason-coded rejections (a policy denial is not a model error), stable-base retraining, replay strata for retention, immutable composite model bundles, a statistical offline gate, a live-shadow hold, and full-bundle rollback. The most valuable property is separation of duties: the training hook can produce an artifact digest but cannot move the active alias — containing training bugs, budget failures, and accidental regressions. The main residual risk is label quality.

Homelab fit

k3s workers run the Rust relay + discovery + API; Redpanda for transactional transport; Redis (persistent) as vault + feedback + registry; NAS for immutable corpora/adapters/rollback bundles; Modal T4 for intermittent, scale-to-zero LoRA training instead of a local GPU; ArgoCD for digest-pinned deploys; Grafana for observability. Training is the one sensible exception to otherwise-local operation — infrequent, digest-addressed, off the latency-critical path.

How the experiment went

The goal: prove the SLM lane can be fine-tuned to correctly infer new schemas with a good score, and that the whole arm-C flow runs on real hardware. The first fine-tune (v1) mode-collapsed — it always said match_schema, missed every new-family and abstain case, and hallucinated schema ids. Five cumulative interventions turned that around (Qwen2.5-0.5B, family-held-out — no sibling of a test family in training):

What each fix did

  • Completion-only loss — mask the long prompt, loss only on the tool-call. The #1 lever; the prompt was drowning the decision signal.
  • Balanced sampling — match_schema was 64% of decisions, so the model could emit new/abstain at all.
  • Finite hypothesis scoring — score every legal completion by likelihood and pick the best, instead of free generation → higher accuracy and out-of-catalog identifiers become impossible (an inference invariant).
  • More data → decision 55% → 78%.
  • Deterministic distance veto — force new_candidate when nothing sits within the gate distance → new-schema recall → 100%.

The key finding

Structural distance cannot cleanly separate a genuinely new family from a drifted/renamed one — both sit at ~0.29 distance. That is a semantic judgment (the model's job, improving with data), while the deterministic layer safely handles the far-and-obvious. The negative result matters more than any headline number: the deterministic lane is necessary but insufficient, and the model has a non-redundant role in the middle. Two honest operating points result — v4 (100% new-recall, conservative) and v3b (78% all-round) — neither universally superior without an explicit error-cost model.

The real-hook proof

The "training ran standalone" gap is now closed for the training half. Honest assessment — solid vs. overclaimed

Established solidly

  • The v1 diagnosis was right — prompt-loss + class imbalance, not parameter count.
  • Finite scoring is a genuine improvement, cleanly isolated (same weights, same inputs).
  • Family-held-out + evaluator-only-sidecar leak-guard is rigorous — kills the earlier tautological-evaluation problem.
  • The negative distance result is honest and important.

Currently overclaimed

  • "External gold" → evaluator-side gold on a synthetic (construction-truth) corpus. Not yet production-feedback validation.
  • The held-out set became a dev/calibration set (inspected across v1→v5). A fresh sealed audit set is required.
  • Figures lack uncertainty — single seed. Need multi-seed, CIs, macro-F1, confusion matrices, recall@k.
  • A deterministic veto's gain is not an SLM gain — report raw / override / gated separately.
  • Zero observed false-merges is not production safety evidence yet.

Documentation drift — fix before any release

The README still describes the SLM lane as unsupported and an older go-live doc calls the gate descriptive-only, while newer code implements trust-application + governed model-registry transitions. Reconcile the README, runbooks, and state diagrams with shipped behavior.

Methodological verdict: rigorous for an early controlled engineering experiment — real LoRA on a real T4, family partitioning, evaluator-side gold, finite closed-set scoring, per-class failure analysis, a deterministic-policy ablation, and an honestly-reported failed distance separator. Not yet a confirmatory benchmark — repeated test-set consultation + single seed + synthetic data keep it exploratory.

What's genuinely next

  • Freeze v3b + v4; build a sealed audit corpus (new families, near-dup clustering, independent seed, no threshold inspection).
  • Multi-seed runs with paired confidence intervals.
  • Complete A1/B1/B2 — the decisive question: does the SLM add safe coverage over a strong deterministic policy at the same risk bound? If B1≈B2, it isn't yet justified.
  • Separate retrieval from semantic accuracy (recall@k, oracle-candidate, end-to-end, gated coverage).
  • Add genuine external data (source-native GitHub Archive / Wikimedia labels, adjudicated homelab feedback).
  • Calibrate operating points into risk-coverage curves, not one threshold.
  • Exercise a complete real trajectory to promotion + forced rollback (training half now done; gate → shadow-hold → promote → rollback remain).
  • Failure drills + poisoning/forgetting validation; reconcile documentation.
