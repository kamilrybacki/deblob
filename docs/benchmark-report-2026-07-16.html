01 Headline

The tagging logic is fast; the exactly-once relay is the bottleneck.

Deblob's actual schema-fingerprint-and-tag work completes in ~0.43 ms and the management/P2-D API answers in single-to-tens of milliseconds. But the Kafka relay wraps every record in its own transaction (begin → produce tagged+discovery → send-offsets → commit), and per-record transactional commits cap end-to-end throughput at ~13.6 records/second — while Deblob's CPU sits at ~0.2 cores and Redpanda at ~0.1 cores. It is latency-bound on the transaction commit, not resource-bound. This is the one thing to fix before any high-volume use.

The load generator can push thousands of messages per second; the relay drains them one transaction at a time. Bars are log-ish scaled for legibility (values labelled).

06 Recommendation

Batch records per Kafka transaction before P3/P4.

The relay's exactly-once guarantee is sound; its granularity is the problem. Committing N records per transaction (instead of one) amortises the ~73 ms commit and should raise relay throughput by roughly two orders of magnitude, while preserving exactly-once semantics per batch. The benchmark proves this is the sole limiter — every other layer is fast:

  • Tagging logic — 0.43 ms/op, not the bottleneck.
  • Redis vault — resolve + publish + P2-D ops all in single-to-tens of ms.
  • P2-D semantic fingerprint + neighbor search — 8–20 ms, ready for use.
  • Resources — the stack is idle under load; no scaling needed, just batching.

Secondary: fix the GET /schemas list SCAN, and finish the bench measurer for end-to-end latency curves. Neither blocks the throughput fix.
