# Continual-Learning / Feedback-Driven Fine-Tuning for SLMs — Joint Research Report
run: jr-slm-cl-161812 · 2026-07-16 · agents: Claude Code + Hermes

## Executive summary
The pattern Deblob built is the industry-standard **"data flywheel"** `[C+H]`: capture production feedback → curate → LoRA/QLoRA fine-tune → held-out + baseline eval → **gated promotion** with rollback. Every source (both agents) converges on this loop. Deblob is well-aligned and has **two genuine differentiators** `[C+H]`: it gates on a **deterministic** zero-false-merge check (no LLM-judge bias), and its trust guards mean even a *promoted-but-bad* model can't corrupt identity — a safety net beyond the gate. **For a 26M–1B function-caller, weighted/stratified SFT (rejection-sampling / RAFT-style) is the right default** `[C+H]`; DPO/KTO applies since accept/reject is preference signal, but there's **no controlled study** on this exact model class + quantized artifact, so method choice needs a *local ablation*, not a citation `[H]`.

The two agents together surfaced **~10 upgrades**, of which the highest-value are: **(1) a bare hash is un-learnable** — store redacted value *shape* (types/ranges/formats/canonical placeholders), not just a hash `[H]`; **(2) a rejection ≠ negative data** — don't treat policy-denials or retrieval-failures as generator-negatives; a rejection needs a *reason*/corrected target `[H]`; **(3) a statistical gate** — "zero false-merge without N" is incomplete; need per-family floors + CIs + non-inferiority margins `[H]`; **(4) feedback-poisoning defense** (reviewer provenance, rate-limit, no single-source auto-retrain) `[H]`; **(5) active-learning curation** (train on the informative, not everything) `[C]`; **(6) catastrophic-forgetting mitigation** (LoRA + replay strata + EWC; retrain from a stable base, not recursive adapter mutation) `[C+H]`; **(7) per-slice gates + composite-bundle versioning + a live-shadow canary ramp + separation of duties** `[C+H]`. `weight=3.0` has no evidentiary basis — ablate it `[H]`.

## Key findings

### Feedback capture & the flywheel
- **The canonical loop is universal** `[C]`: capture (input+output+context) → filter/rank → curate → fine-tune (LoRA/QLoRA) → eval vs held-out + baseline (automated first, human for promising) → promote winners + rollback. (tianpan.co, theneuralbase, Continuum, tohio/data-flywheel, NVIDIA NVInfo AI/ACL-2026.)
- **"Confidence ≠ correctness" is THE flywheel failure mode** `[C]`: the danger is the model being *confident-and-wrong* while getting zero feedback → you never train on it → the flywheel "amplifies confident mistakes." Mitigation: a correctness-check layer + sample high-confidence outputs for spot-checks. **→ Deblob avoids this by construction** — it uses *deterministic corroboration* (not model confidence) and captures *human rejections* (exactly the confident-and-wrong cases a human caught). Deblob's `TrustedProposalRejected` hard-negative is precisely the highest-value signal the literature says is hardest to get. (theneuralbase, tianpan.co.)
- **Correction-as-preference** `[C]`: the diff between a model output and the human-edited/-chosen alternative is a fine-grained preference pair (GitHub Copilot harvests this at scale). Deblob's accept/reject is this signal.
- **Quality over quantity + active learning** `[C]`: "a well-curated 5,000-example set beats 500,000 uncurated." Train on the *informative* examples — uncertain, near-decision-boundary, underserved distributions — not everything. `<1%` of interactions yield explicit feedback; naive training on it causes sycophancy/survivorship bias. (tianpan.co, Zylos, theneuralbase.) **→ Deblob GAP:** captures all feedback but does not yet curate/prioritize by informativeness.
- **NVIDIA NVInfo AI (real deployment)** `[C]`: 3-month HITL flywheel, 495 negative samples → fine-tuned an 8B to replace a 70B router (96% acc, 10× smaller, 70% faster). A repeatable HITL blueprint; small curated negative sets are enough.

### Method for tiny function-call models
- **RAFT / rejection-sampling FT is a surprisingly strong, simple baseline** `[C]`: "outperforms or matches PPO and iterative DPO" — SFT on positively-rewarded (accepted) samples; Reinforce-Rej filters both all-correct and all-incorrect. Ideal for a tiny model + limited compute. (arxiv 2504.11343 "Minimalist Approach"; modelscope/ms-swift RFT.)
- **DPO / RS-DPO / KTO** `[C]`: reward-free preference optimization, stable, lightweight, no separate reward model — good in limited-resource settings; RS-DPO (rejection-sampling + DPO) beats RS/PPO/DPO. Applicable to Deblob since accept/reject forms preference pairs. **→ Deblob GAP:** captures examples individually, not as (chosen, rejected) pairs — add pair formation to enable DPO/KTO. (AWS ML blog; RS-DPO NAACL-2024; DPO paper.)
- **On-device continual learning = LoRA adapters + replay buffer + EWC** `[C]`: the consensus anti-forgetting stack. K-Merge (online adapter merging for on-device 1–2B LLMs), MSSR (memory-aware adaptive replay), baseten rank-1 LoRA (hot-swap adapters, "ring buffer of recent deltas for rollback," serve-with-adapter + shadow-replica-merge), and LoRA+lightweight-replay for streaming. Caveat (Biderman 2024): **LoRA forgets less but also learns less** — a capacity/retention trade-off. **→ Deblob GAP:** the external fine-tune hook + orchestrator should provide a **replay set** (prior examples) and use LoRA+EWC; Deblob currently exports only new feedback + synthetic, no explicit anti-forgetting replay.

### Governed / gated model promotion (Deblob's ModelRegistry validated)
- **The registry's value IS the gate** `[C]`: immutable versioned artifacts + aliases (champion/challenger/shadow) + promotion requiring *evidence* (eval results, model card, sign-off) + rollback by repointing the alias. (MLflow Model Registry, datarekha, jellewillekes/ml-lifecycle-platform.) **→ Deblob matches this: immutable versions, gated promotion, audit, rollback, "beats current active."**
- **Gate on per-slice + calibration, not just aggregate** `[C]`: 9shrey/cicd-retraining gates on AUC + calibration/ECE + **per-slice metrics**; ModelGate runs 8 checks (regression tolerance, absolute floor, fairness DP/EO, drift PSI, "strictly better UNLESS champion older than 14 days"). **→ Deblob GAP:** its gate is aggregate — add **per-family/per-slice gates** (no ≥N-example slice regresses) and a **stale-champion override** so a static active model can't stall improvement.
- **Rollback must be a pointer flip + dry-run** `[C]`: "rollback must be instantaneous (traffic weight flip, not redeployment)"; ModelGate's `rollback --dry-run` is "non-negotiable." **→ Deblob has rollback; add a dry-run/plan mode.**
- **Progressive rollout: shadow → canary → blue-green** `[C]`: promotion to an alias ≠ full traffic; ramp 5/25/50/100% with guardrail metrics + automated rollback on SLO breach. (CalibreOS playbook — Uber/Netflix/Google; 9shrey pipeline.) **→ Deblob GAP:** promotes straight to *active*. Since the SLM lane is **already shadow-only**, the natural fit is: a newly-promoted model first runs in **live shadow** (gathering real eval) for a hold period before its trust-policy authority expands — a canary without user-facing risk.
- **Reproducibility/lineage bundle** `[C]`: capture dataset fingerprint + config hash + model digest + git SHA per version (jellewillekes). **→ Deblob:** has `trained_from`; expand to a full lineage bundle.

## Conflicts & adjudication
- *DPO vs PPO vs RAFT* — sources disagree on the best preference method in general (Rafailov favors DPO; Ivison/OpenReview find PPO edges DPO on reasoning; the "Minimalist" paper finds RAFT matches both). **Adjudication for Deblob's tiny FC task:** start with RAFT-style weighted SFT (simplest, strong, stable on small models), add DPO/KTO only if preference-pair volume justifies it — the task is arg-extraction, not open-ended reasoning where PPO's edge shows. `[C]`

## Hermes' perspective (design review — the high-value half)
Hermes corroborated the flywheel/method/promotion findings `[C+H]` (KTO for binary/unpaired feedback; *"LoRA Learns Less and Forgets Less"*; replay-based I-LoRA / O-LoRA / On-Policy-Replay; the autonomous-driving closed-loop survey as the canonical triggered-collection→shadow→relabel→regression→release pattern; MLflow/SageMaker/Vertex registries with approval status + rolling deploy + rollback). Her vault notes on Cactus-Needle, FunctionGemma-270M, and Granite-4.0-Nano confirm: **capture on-device, but train offline + governed; weighted/stratified SFT is the best default for a tiny bounded function-caller with known-correct outputs; LoRA/QLoRA > full for this class** (architecture/dataset-dependent).

Her **review of Deblob's design added several items my web sweep missed** — the most important listed here `[H]`:

1. **A hash alone is un-learnable (the biggest fix).** "No raw values" is privacy-safe, but argument-learning needs *value shape/semantics* — store redacted synthetic substitutes, structural types, ranges, formats, and canonicalized placeholders, not just a hash. A bare hash can neither reproduce training nor diagnose an argument failure.
2. **A rejection is not automatically negative training data.** Add a **rejection reason** + a corrected target where possible. **Do NOT treat a policy-denial or a retrieval-failure as generator-negative data** — the model may have been right and the *retrieval/policy* wrong; mislabeling poisons the set. A rejection without a corrected alternative isn't valid SFT/DPO data on its own.
3. **Feedback poisoning defense.** Record actor/source, trust level, tool/schema version, review reason, dedup cluster; rate-limit/quarantine anomalous sources; **never let one reviewer or one noisy family trigger automatic retrain + promotion.**
4. **Statistical gate, not just "zero."** "Zero false merges *without N* is incomplete." Define minimum test counts, per-family floors, confidence intervals, and non-inferiority margins.
5. **Separation of duties.** The training hook registers a *candidate only*; evaluation writes gate evidence; a *separate* controller / explicit approval moves the champion alias.
6. **Composite artifact bundle.** Version tokenizer + prompt + runtime + quantization + retrieval/index + grammar + catalog *with* the weights; roll back the whole bundle. Evaluate the *quantized* artifact separately from the training checkpoint.
7. **Retrain from a stable base, not recursive adapter mutation.** Rebuild each release from a reproducible snapshot; don't sequentially mutate the latest adapter.
8. **Redis is the capture queue, not the system of record.** Export immutable, content-addressed dataset snapshots (manifests + checksums) to NAS/MinIO; keep a **never-trained-on** final safety suite; split by source/time/near-dup-cluster/family so a paraphrase can't cross the train/test boundary.

Homelab-fit pattern she proposes: Redis (capture + dedup index) · NAS/MinIO immutable paths (datasets/adapters/weights/reports) · SQLite/Postgres or OSS MLflow (lineage/aliases/audit) · digest-pinned deploy manifest · champion+challenger pods where memory permits · sampled shadow replay · alias/traffic-split changed independently of the model build · previous champion retained for instant rollback. *"A small append-only registry table + immutable artifacts is sufficient if the state machine and manifests are rigorous."*

Her confidence flags: `weight=3.0` has **no evidentiary basis** — keep it an ablated hyperparameter with a cap on correlated-rejection dominance; native llama.cpp training is WIP (don't assume production-ready); continual-learning research is mostly on models ≫ Needle, so forgetting results may not transfer; there's **no controlled study** comparing weighted-SFT / DPO / KTO / RS-SFT on the same 26M–1B FC benchmark + quantized artifact — so the method choice needs a local ablation, not a citation.
