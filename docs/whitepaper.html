Deblob: Emergent Schema Identification for Event Streams
via a Deterministic Structural Core and a Trust-Gated Language-Model Lane

Kamil Rybacki

Technical whitepaper · July 17, 2026 · github.com/kamilrybacki/deblob

Status: pre-alpha research prototype — all learning results are development-set estimates.

Abstract

Event streams accumulate schema "blobs": producers emit JSON whose structure drifts, forks, and multiplies until no one can say how many distinct schemas a topic actually carries. Deblob tags every event with the schema family it belongs to and discovers genuinely new families, with authority split between two layers: a deterministic structural core that tags events in 0.43 ms inside an exactly-once Kafka relay, and a small fine-tuned language model whose proposals can never be applied without independent deterministic corroboration. Fine-tuned on 720 synthetic labelled examples for an observed GPU charge under $0.10, the combined decision policy identifies never-before-seen schema families at 89% recall on family-held-out cases, never once assigning one to an existing schema; finite-hypothesis scoring restricts the model to catalogued identifiers by construction. Because the held-out set was consulted across six experiment variants, these are development-set operating-point estimates, not sealed confirmatory results. We report the architecture, data representation, training method, an instructive negative result about structural distance, and honest limitations.

Keywords: schema inference, event streams, exactly-once processing, small language models, LoRA fine-tuning, selective prediction, human-in-the-loop

1 Introduction

A production event bus is rarely the tidy place its documentation claims. Teams ship new producers, rename fields, wrap payloads in envelopes, and re-use generic field names (id, status, amount) for unrelated things. The result is a stream whose effective schema catalogue is unknown: a blob. Downstream consumers break on drift they cannot see; data teams cannot answer "how many event types do we actually have?"

Two definitions anchor everything that follows. A schema family is a governed identity grouping the structurally evolving representations of one logical event type. A false merge assigns events from two logically different types to the same family — the costliest possible error, because it silently poisons every downstream consumer, whereas a deferred decision merely costs a human review.

Deblob decides, for every event, one of three things: it matches a known family (and how: exactly, as compatible drift, or as a structurally similar but semantically incompatible lookalike), it is a new family (emitted as the action new_candidate; we use the terms interchangeably), or the evidence is insufficient and the system abstains, deferring to a human. In practice the flow looks like this:

Input: JSON event whose structural shape was seen before Result: exact fingerprint match → tagged deterministically, 0.43 ms Input: similar profile, fields renamed by a vendor prefix SLM: proposes "candidate 2, compatible drift" Gate: applies it only if rank, distance, margin, and evidence all corroborate — otherwise → human review

The outputs are concrete: every event republished with its family tag, a continuously maintained schema catalogue, and a short review queue holding only the cases no layer could decide safely. This paper contributes:

  1. a two-lane architecture that separates deterministic identity authority from learned semantic proposals;
  2. finite candidate scoring combined with an independent deterministic trust gate; and
  3. a controlled fine-tuning experiment, including a negative result showing that structural distance alone did not separate new families from drifted ones on our corpus.

Related work. Deblob composes established parts rather than inventing them: schema registries and schema-evolution tooling govern declared schemas but assume producers register them; exactly-once stream processing via Kafka transactions provides the delivery substrate; selective prediction [3] motivates abstention as a first-class output; and candidate-constrained language-model inference underlies the scoring scheme of §5.3. The contribution is the composition of authority — deterministic identity, learned proposals, gated application — not any individual component.

2 System Design

2.1 Deterministic core

Events are parsed with bounded, fuzz-hardened parsing, reduced to a canonical structural fingerprint (key order, whitespace, and value content do not affect it), and folded into a per-family monoid profile — a summary of field paths, types, nullability, and observation counts that can be merged associatively across batches, so event history never needs re-scanning. Candidate families are retrieved by structural distance, an overlap-based measure (weighted Jaccard) over the profile where lower is more similar; the margin is the runner-up's distance minus the top candidate's — larger means less ambiguous. The relay consumes from Kafka and republishes tagged events under exactly-once transactional semantics (a crash or retry can never emit a duplicate or drop an event); batching plus intra-batch produce pipelining raised throughput 43× (from 14 to 598 records/s) with the exactly-once guarantee preserved per batch and verified under crash, rebalance, and duplicate-delivery chaos tests. Published schemas live in a Redis vault with atomic, write-once publication.

2.2 The trust gate

The SLM lane may only propose. A proposal to match an existing family is applied only when the deterministic evidence independently corroborates it: the proposed family is the top-ranked retrieval candidate, its structural distance is at most 0.15, the margin to the runner-up is at least 0.10, and the family has at least 20 observations. Model confidence plays no role. Everything else routes to a human. The precise guarantee, verified by an exhaustive test over all gate-condition combinations, is: an SLM proposal can never be applied without deterministic corroboration. That is a structural property of the gate — it does not make the corroboration itself semantically infallible, which is why empirical false-merge rates are reported separately (§4).

2.3 Governed continual learning

Human decisions (promotions, accepted and rejected proposals) are captured as immutable, provenance-bearing training examples, with reason codes so a rejection caused by a policy denial is not mislabelled as a model error. Retraining runs remotely from a stable base checkpoint; a candidate model must pass a statistical offline gate (zero false merges on a held-out corpus, per-family precision floors with confidence bounds, non-inferiority versus the active model) and a live-shadow hold before an explicit, separate promotion step activates it. The training hook returns artifact digests only; by construction it cannot move the active-model alias. Full-bundle rollback (weights, tokenizer, prompt template, quantization, retrieval index) is retained.

3 Input Data and Schema Representation

Deblob ingests ordinary JSON events. A representative GitHub Archive event:

{ "id": "2489651045", "type": "PushEvent", ← never shown to the model "actor": {"id": 91, "login": "octocat"}, "repo": {"id": 3, "name": "octocat/hello-world"}, "payload": { "push_id": 536390, "size": 1, "ref": "refs/heads/master", "head": "0e5bfe237a11f2c9…" }, "created_at": "2015-01-01T00:00:00Z" }

The model never sees raw events. Each event contributes to a redacted structural profile: field paths, types, presence and null counts, and numeric range buckets — no values. The decision input concatenates that profile with the retrieved candidates and their structural distances:

observation_count: 54 field_count: 6 path=["calibration"] types=["object"] present=54/54 path=["calibration">"offset"] types=["number"] numeric=["11-100"] path=["calibration">"unit"] types=["string"] path=["readings"] types=["array"] path=["sensor_reading_id"] types=["string"] RETRIEVED CANDIDATES (structural distance only; lower = closer) rank=1 family=fam_b128… schema=sch_pfxy… distance=0.286 rank=2 family=fam_8c91… schema=sch_hlxg… distance=0.444 ALLOWED schema_id SET = [sch_hlxg…, sch_pfxy…]

This representation has two roles. It is a privacy boundary: the profile renderer excludes scalar values and known label-bearing fields from model inputs, and tests enforce that boundary for the covered rendering paths — though field names and aggregate buckets may still carry sensitive information and require separate policy controls. It is also the learning target: the shape statistics carry exactly the evidence a schema decision needs. Label-revealing fields (type, $schema, topic names) are stripped before inference and kept in an evaluator-only sidecar, so evaluation gold can never leak into the input.

4 Deterministic Core: Measured Performance

Retrieval alone, however, is an unsafe standalone decision rule: accepting every top-1 candidate on the benchmark corpus yields 100% coverage with 39 false merges in 96 events. Passing the same events through the trust gate cuts coverage to 54% and, on this benchmark, the configured gate produced one observed false merge among accepted decisions (95% upper bound 10.1%), with the remainder deferred to a human. That trade — coverage for a hard safety posture — is the system's operating philosophy: what the gate cannot corroborate becomes a safe error (a review), not a dangerous one (a merge).

5 Learning the Semantic Decision

5.1 Task and corpus

The SLM's task is a bounded classification rendered as a tool call: given the redacted profile and candidates, emit match_schema (with a relation: exact, compatible_drift, or incompatible_similarity), new_candidate, or abstain. One combination deserves explanation: match_schema with incompatible_similarity means the model selected a retrieved catalogue candidate as the relevant comparison and judged that it must not be merged — an accepted-match flag of "no". Training and evaluation use a synthetic corpus whose labels are true by construction: 90 base families, each expanded by known transformations — 30% exact re-encounters, 20% compatible drift (added optional fields, widened nullability, wrappers), 10% incompatible lookalikes (unit swaps, generic-name re-use: the false-merge traps), 20% new families, 20% ambiguous or low-evidence cases (900 cases). The split is family-partitioned: all variants of a family land on the same side, so a held-out family's siblings never appear in training (720 train / 180 test).

5.2 Training method

Two training choices carry most of the improvement. First, completion-only loss: the decision prompt is hundreds of tokens and the answer roughly thirty; under a naive full-sequence objective, more than 90% of supervised tokens belonged to the prompt, allowing prompt reconstruction to dominate the sequence-level objective — and the first attempt duly collapsed to always answering match_schema (0/16 new families detected). Masking the prompt concentrates all learning on the decision. Second, balanced sampling: matches dominate the natural distribution (60/20/20), so minority decisions are oversampled to parity before training.

5.3 Finite-hypothesis scoring

At inference the model does not generate freely. The legal decision space is finite — each allowed schema identifier crossed with three relations, two novelty labels, three abstain causes — so Deblob enumerates every legal completion, scores each by length-normalised conditional log-likelihood under the model, and selects the argmax. On identical weights this nearly doubled decision accuracy versus free generation (30% → 55%) and enforces identifier validity by construction: an identifier outside the retrieved catalogue can never be selected — though the model can still select the wrong legal identifier. A deterministic veto completes the policy: if the nearest known family is beyond the gate distance, the decision is forced to new_candidate regardless of the model's preference.

6.4 Structural distance did not separate new from drifted

This is the paper's central negative result: on this family-partitioned synthetic corpus, no tested single distance threshold cleanly separated new families from compatible drift. A pure distance rule at 0.15 catches every new family but wrongly splits drifted ones; widening it to 0.40 (§6.3, v5) reverses the trade. We read this as evidence — not proof over all schema populations — for the layered policy: the veto handles distances beyond plausible drift, the fine-tuned model judges the middle, and the trust gate turns whatever either gets wrong into a deferral rather than a merge.

6.5 End-to-end training-path verification

The remote-training half of the governed loop was exercised end-to-end on the deployed homelab infrastructure: a feedback snapshot was exported with a content-addressed manifest; the deployed remote trainer resolved it, trained the adapter of Table 2, and returned artifact digests only; the adapter was reloaded from its published artifact and re-evaluated, reproducing the evaluation results (new-family recall 100% on the 120-case validation subset; 88.9% on the full 180-case set, Table 3b). The statistical gate, live-shadow hold, and explicit promotion are implemented and tested separately (against real Redis) but were not part of this run — promotion remains a separate, gated controller operation that the training hook, demonstrated in the same run, cannot perform.

7 Limitations

  1. Synthetic ground truth. Evaluation gold is independent of the gate and hidden from the model, but it is generated by construction, not collected from production traffic. Ingestion for real corpora (GitHub Archive, Wikimedia event streams) exists; sealed evaluation on them has not yet run.
  2. The held-out set became a development set. Six variants consulted the same held-out families, so the reported figures are optimistically selected operating-point estimates. A sealed audit corpus with frozen thresholds, and multi-seed runs with confidence intervals, are the required next step.
  3. Single seed, point estimates. All accuracy figures are one training trajectory.
  4. Attribution is explicit but asymmetric. As Table 3a shows, the combined policy's new-family recall is supplied entirely by the deterministic veto; the model alone reaches 33.3%. Neither configuration dominates: the model supplies the match and abstain discrimination the veto cannot.
  5. Truncation may have skewed the training distribution. The 640-token cap retained only 421 of 1,296 oversampled rows, and post-truncation class and family balance was not audited. Future rounds should compact prompts or preserve complete targets under left truncation.
  6. One observed false merge is not a production safety bound. The gate's guarantee is conditional (no application without corroboration); its empirical false-merge rate on real traffic is unmeasured. Production-scale labelled shadow traffic is needed for a production-grade bound.
  7. Pre-alpha software. Single maintainer, no release history; some documentation still describes earlier states of the system.

8 Conclusion

Deblob splits schema identification into a fast deterministic core and a small, cheaply trained language model whose authority is bounded by construction: the model proposes, the deterministic layer corroborates, and nothing is applied without that corroboration. A controlled experiment showed the decision is learnable by a 0.5-billion-parameter model, showed that inference formulation (finite candidate scoring) matters as much as training, and produced the negative result that structural distance alone could not separate new families from drifted ones — the clearest evidence that the learned lane has a non-redundant role. The continual learning loop, from feedback capture through remote training to governed promotion, is implemented and its training half verified on deployed infrastructure. The architectural claim this work supports is narrow and, we believe, the important one: learned coverage can increase without transferring schema-identity authority to the model.
