# Deblob P2-D (Semantic Fingerprint) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Add the third identity dimension — a deterministic `sem_` fingerprint over controlled-vocabulary field metadata — so byte-identical structures with different meaning get distinct identity, without changing the wire tag, without SLM auto-application, and without ever causing a merge.

**Architecture:** A new pure `deblob-semantic` crate owns the `deblob-semantic-v1` vocabulary tables, validation, canonical serialization, and the digest. `deblob-core` gains the `sem_` id + metadata types + two optional fields on `SchemaRecord`. `deblob-redis` stores the semantic block on the immutable schema hash via the existing atomic Lua path (byte-compare immutability, non-destructive migration). The `deblob` bin exposes an authenticated+audited governance API and a computed (not-acted-on) semantic-drift signal. Un-annotated schemas are byte-identical to today.

**Tech stack:** Rust; `sha256`/`data-encoding` (reuse the `digest_id!` macro), `serde` with `deny_unknown_fields`, the existing `deblob-*` crates, `redis`. No new heavy deps.

**Spec:** `docs/superpowers/specs/2026-07-15-deblob-p2d-semantic-fingerprint.md` — referenced by § below.

## Global constraints

- **`sem_` NEVER rides on the wire in P2-D.** No `deblob-semantic-id` header. Message tagging stays `sch_`/`cand_`. A test must assert the wire tag is unchanged by annotation.
- **Never free prose.** Every stored/hashed semantic attribute is a registered, versioned vocabulary token. No free-string field exists in `FieldSemantics`.
- **Empty metadata → `SemanticFingerprint::None`**, never a hash of the empty map. Un-annotated = semantically *unknown*, never semantically *equal*. Two un-annotated schemas must never compare as the same semantic identity.
- **Only ever ADD a split.** The semantic fingerprint can never cause two families to merge. Semantic-drift is a one-directional split signal.
- **Operator-gated, like promotion.** Semantic metadata enters only via the authenticated + audited management API. Producer-supplied semantic input on the ingest path is stripped. The SLM (shadow) only proposes; it never applies.
- **No auto family-split.** Semantic-drift is computed + surfaced only. Acting on it is the P3 live gate.
- **Immutability + non-destructive migration.** Stored `sem_`/metadata is byte-compared on re-publish (mismatch fatal, like `canonical`); pre-P2-D records read back `None`; `rebuild_index` preserves `sem_`.
- **Off by default / no-op.** Un-annotated behavior is byte-identical; every existing test stays green.
- Domain separation: `sem_` uses `deblob-semantic-v1`, distinct from `sch_`'s `deblob-canon-v1`; a `sem_` can never be parsed as a `sch_`.
- TDD; 80%+ coverage; deterministic golden tests for the digest.

### Task 3: canonical serialization + the `sem_` digest

**Files:** create `crates/deblob-semantic/src/{canon.rs, digest.rs}`; `crates/deblob-semantic/tests/digest_it.rs`.

**Produces:** `canonical_semantic_bytes(&SemanticMetadata) -> Vec<u8>` — field paths sorted by code point; within each field a fixed attribute key order; `enum_semantics` entries sorted by enum value (code point); absent attributes omitted (not emitted as null); minimal + stable encoding. `semantic_fingerprint(&SemanticMetadata) -> SemanticFingerprint` — if the map is empty (no field carries any attribute) → `None`; else `Some(SemanticId::from_digest(&sha256("deblob-semantic-v1\0" ++ canonical_semantic_bytes)))`.

### Task 4: field-path ↔ structure consistency

**Files:** `crates/deblob-semantic/src/lib.rs` (a `paths_of_canonical` helper or reuse a `deblob-fingerprint` path enumerator); a `validate_paths` fn.

### Task 5: vault storage — atomic annotation, immutability, migration, reverse lookup

**Files:** `crates/deblob-redis/src/registry.rs`, `crates/deblob-redis/src/lua.rs`; reuse the schema-hash + audit + rebuild paths.

**Produces:** `annotate_semantic(sch_id, metadata, sem_fp, actor, reason)` writing the semantic block + recomputed `sem_` + an audit event through **one atomic Lua transition** (all-or-nothing). The stored semantic block is byte-compared on any conflicting write: an idempotent replay (identical block) is a no-op; a *different* block is a governed change that requires a reason and writes a new audit event, while a re-*publish* of the schema with mismatched stored semantics is **fatal** (same treatment as a `canonical` mismatch). `read_semantic(sch_id) -> Option<(SemanticMetadata, SemanticId)>`. `schemas_by_semantic(sem_id) -> Vec<SchemaId>` reverse lookup (a `deblob:sem-index:<sem_id>` set, maintained in the same Lua transition). Non-destructive migration: a schema record without the fields reads back `None`; `rebuild_index` restores `sem_` bindings + the reverse index from the schema records.

- [ ] Write failing tests (vs a real Redis, mirroring the existing registry IT harness): annotate a schema → `read_semantic` returns the metadata + `sem_`; `schemas_by_semantic` finds it; re-annotate with a *different* metadata (with reason) → new `sem_` + a new audit event; a conflicting schema re-publish with mismatched stored semantics → fatal error; a legacy record (no semantic fields) reads back `None`; after `rebuild_index`, `sem_` + reverse index are restored.
- [ ] Run → implement → run.
- [ ] Commit `feat(redis): atomic semantic annotation, byte-compare immutability, migration, sem_ reverse index`.

Reviewer checkpoint — vault immutability + migration + atomicity are the corruption surface.

### Task 7: semantic-drift signal (computed, not acted on)

**Files:** create `crates/deblob/src/semantic_drift.rs`; wire into the family-version + metrics paths.

**Produces:** when a family gains a **structurally-compatible** new version (per the P1 drift policy) whose annotated `sem_` differs from the prior version's `sem_`, record a `SemanticDrift { family_id, prior_version, new_version, prior_sem, new_sem }`: a Prometheus counter `deblob_semantic_drift_total`, a field on the family-version record, and an API-queryable entry. It **does not** split the family. One-directional: it flags compatible-versions-that-mean-different-things (a split candidate), never a merge.

- [ ] Write failing tests: two compatible versions with the *same* `sem_` → no drift; two compatible versions with *different* `sem_` → the counter increments, a `SemanticDrift` record is emitted, and the family is **not** split (version count/family id unchanged); an un-annotated → annotated transition (prior `None`) is handled explicitly (documented: `None`→`Some` is a first-annotation, not a drift, per the runbook).
- [ ] Run → implement → run.
- [ ] Commit `feat(bin): semantic-drift signal — computed and surfaced, never auto-splits`.
