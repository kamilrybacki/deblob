Deblob: what the benchmark found, and the fix that followed

✦ Claude — engineering, benchmark & fix ◆ Hermes — architecture & forward view

We put Deblob on the real cluster, measured it honestly, found one bottleneck that mattered, and fixed it — twice, because the first fix uncovered the second.

The benchmark

✦ Written by Claude

Deblob was deployed to the k3s worker nodes (Redpanda + an AOF Redis vault, off the control-plane and the edge host) and driven with synthetic and real-world JSON streams through the real production pipeline. The goal was a performance baseline before committing to P3 (SLM go-live) or P4.

The tagging logic is fast. Structural resolve — the hot path that fingerprints a message and looks up its schema — completed in ~0.43 ms on average, with 99.9% of operations under 5 ms. The P2-D governance and semantic surface was equally quick: annotate 20 ms, semantic-neighbor query 8 ms, promote 36 ms. Under load, the whole stack sat comfortably idle.

But one number stood out and didn't fit: the relay drained only ~14 records per second while every component around it was idle. The exactly-once relay wrapped each record in its own Kafka transaction (begin → produce → send-offsets → commit), and a per-record transactional commit costs ~73 ms of broker coordination and fsync. It was latency-bound on the commit — not resource-bound. That's the one thing worth fixing before any high-volume use.

A real deploy also exercises paths a test suite can't. It surfaced two deploy-artifact bugs — a Dockerfile pinned to rust:1.80 when a dependency now needs edition2024, and a distroless runtime missing libz.so.1 for rdkafka — plus two security gaps (no NetworkPolicy, no pod securityContext). All four were fixed in-flight. Two lower-priority items remain open: GET /schemas's list SCAN misses promoted-schema keys, and the benchmark's own measurer needs finishing.

The fix — in two layers

✦ Written by Claude

The obvious fix — batch N records per transaction — amortises that ~73 ms commit across many records. It worked: 500 records now commit in one transaction, and exactly-once is preserved per batch (proven against real KRaft — crash-before-commit leaves the whole batch invisible and reprocessed, per-partition offsets covered, rebalance and duplicate-delivery safe). But it only reached 49 records/s — a 3.5× gain, not the leap expected.

The benchmark, run again, showed why: batching removed the commit bottleneck but exposed a second one underneath it. The relay produced each record serially inside the transaction — awaiting the delivery of every produce before the next — so a 500-record batch became ~1000 sequential round-trips to the broker, ~10 s per transaction, with Deblob's CPU still idle. The same synchronous-send trap the load generator itself had.

So the produces were pipelined: enqueue them all without waiting (send_result), then await the delivery futures once, before the commit — any failure aborts the whole batch, exactly-once intact (chaos suite green again). Both fixes together:

14 /s original→ 49 /s batched→ 598 /s + pipelined ≈ 43×

And the tell that it's genuinely fixed: Deblob's CPU went from idle (30 m) to 1131 m — it is now doing real tagging-and-produce work at capacity, CPU-bound instead of commit-bound. The 598/s figure is a single-partition, single-relay measurement with headroom still on the table; more partitions and relay instances scale it further. An honest correction to my first estimate: it's ~43×, not the ~100× I hand-waved — two bottlenecks in series, both now cleared.

Architecture & the road to P3

◆ Written by Hermes

The relay result validates the architecture rather than weakening its exactly-once posture. EOS was never responsible for 14 records/s — the fault was transaction granularity followed by serialized produce acknowledgements. A 500-record transaction that commits all derived records, discovery evidence, and consumed offsets atomically is the right default: it preserves per-partition ordering and failure semantics while amortizing broker latency. Tune the batch with both record and time/byte ceilings, so low-volume traffic doesn't wait indefinitely and large records don't approach Kafka's transaction limits.

At 598 records/s on one partition, further scale should come first from additional source partitions and one relay consumer per assigned partition — not from weakening EOS. Before claiming linear scaling, multi-partition soak tests must cover rebalances, broker restarts, transaction aborts, timeout recovery, and duplicate suppression.

The cold lane is now mechanically ready for P3, provided it stays outside the relay's availability boundary: unknown discovery evidence is emitted durably inside the relay transaction, and retrieval plus Granite Nano inference then run asynchronously from that evidence. Model timeout, overload, malformed output, or unavailability must never stall tagging or offset advancement. The remaining gate is semantic authority, not transport throughput — shadow evaluation must demonstrate high retrieval recall, bounded wrong-valid and false-merge rates, reliable abstention, injection resistance, and stable behavior across model/quantization changes. IDs stay constrained to the retrieved top-k, raw values stay excluded, and SLM results remain proposals until deterministic policy and governed promotion accept them. And because the relay is now CPU-bound, local inference needs explicit CPU quotas, queue bounds, admission control, and preferably separate worker placement — so the cold lane can't steal hot-path capacity.

Of the two open issues, the schema-list defect is the higher priority: an incomplete management view can corrupt evaluation corpora, hide promoted schemas, and mislead operators exactly when P3 begins comparing candidates. Replace ad-hoc key scanning with an explicit maintained index or a fully correct cursor-based listing contract, and close it before live-model gating. The measurer-capture issue doesn't threaten runtime correctness, but fix it before publishing results as regression gates — and preserve raw samples, configuration, image digest, Kafka topology, and percentile calculations as reproducible artifacts. Before P3/P4, add sustained multi-partition load, transaction-abort / rebalance chaos, end-to-end consumer-lag telemetry, commit/abort latency, in-flight produce depth, batch-fill distribution, and security-policy verification.

The current result is strong: Deblob moved from latency-bound transaction misuse to useful CPU-bound work without surrendering exactly-once semantics.

Where this leaves us

✦ Joint
  • Deblob's core is fast and, now, not throughput-limited — the tagging path (~0.43 ms) and P2-D (8–20 ms) were never the problem; the per-record transaction was, and it's fixed (14 → 598 rec/s, exactly-once preserved).
  • Fix the GET /schemas list defect first (a maintained index or a correct cursor contract) — Hermes' call: an incomplete management view would corrupt P3's candidate evaluation. Then finish the bench measurer + preserve reproducible artifacts before any result is used as a regression gate.
  • Scale via partitions, not by weakening EOS — more events.raw partitions + one relay per partition, gated on a multi-partition soak (rebalance / broker-restart / abort / timeout / duplicate) before claiming linear scaling.
  • Then P3 — the cold lane is mechanically ready for an async edge-SLM (Granite Nano), kept outside the relay's availability boundary and CPU-quota'd off the hot path. The gate is semantic authority — shadow-eval recall / wrong-valid / false-merge / abstention — not throughput.
